# Backups

Applies to **mode 2 (LAN beta)** and **mode 3 (VPS)** in
`docs/deployment.md`. Mode 1 (local-only development) doesn't need
backups — wipe with `docker compose down -v` whenever.

> **Single-node beta caveat.** Backup discipline is the only thing
> standing between you and total data loss. There is no replica, no
> automated DR. **If you skip the off-host copies and the monthly
> restore drill, you don't have backups — you have a hope.**

## What to back up

| Source                                    | Captures                                                                                       |
| ----------------------------------------- | ---------------------------------------------------------------------------------------------- |
| Postgres (`pg_dump --format=custom`)       | users, sessions, scenarios, briefs, artifact metadata, questions, answer keys, indicator sets, scenario progress, question responses |
| `/var/lib/citrain/artifacts/` (VPS) or the named `artifacts` volume (LAN) | the actual artifact bytes referenced by the DB                                  |
| `deploy/env/vps.env` (VPS) or `deploy/env/local.env` (LAN) | DB credentials, seed config, `TRUST_PROXY`                                          |
| `/etc/caddy/Caddyfile` (mode 3, optional mode 2) | reverse-proxy config; trivial to recreate                                                |

The first two are the irreplaceable pair — a Postgres dump without
the artifact directory leaves every `artifacts.sha256` value
referencing a file that's no longer on disk.

> **M9 reminder.** Admin-uploaded artifacts (added via the
> `/admin/challenges/:slug/edit` UI from M9 onward) live in the same
> `scenarios/<scenarioId>/<uuid><ext>` tree the seed writes to.
> Backing up the `artifacts` directory captures **both** seeded and
> admin-authored content; no separate workflow is needed.

## What NOT to back up

- The Docker image cache. Rebuild from `git pull && docker compose build`.
- `node_modules`. Rebuild from the committed lockfile.
- Caddy's autogenerated TLS certificates. Caddy refetches on demand.

## Storage layout (mode 3)

```
/var/lib/citrain/
├── postgres/        (live database files — DO NOT tar this; use pg_dump)
├── artifacts/       (live artifact bytes — tar this directly)
└── backups/
    ├── postgres/    (pg_dump output, one .dump file per nightly run)
    └── artifacts/   (tar.zst snapshots of /var/lib/citrain/artifacts)
```

Mode 2 (LAN beta) uses Docker named volumes by default. The script
below detects the right path automatically by talking through
`docker compose exec` for the dump, and through a temporary container
for the artifact tar. If you'd rather have host bind mounts in mode 2
too, edit `deploy/docker-compose.local.yml` to mirror the VPS file's
`volumes:` paths.

Backups live on the host that runs the stack initially — then
**immediately get copied off-host** by a separate cron job. A backup
that lives only on the box you're backing up isn't a backup.

## Nightly backup script (mode 3)

`scripts/backup-vps.sh`:

```bash
#!/usr/bin/env bash
# ci-train nightly backup (mode 3 — VPS).
#
# Dumps Postgres + tars the artifact directory under
# /var/lib/citrain/backups/. Retention: 14 dailies.
#
# Off-host replication is a separate cron entry below — do NOT skip
# that step. A backup that lives only on the box you're backing up
# isn't a backup.

set -euo pipefail

REPO=/opt/citrain
ENV_FILE=$REPO/deploy/env/vps.env
COMPOSE_FILE=$REPO/deploy/docker-compose.vps.yml
BACKUP_DIR=/var/lib/citrain/backups
TS=$(date -u +%Y%m%dT%H%M%SZ)

COMPOSE="docker compose --env-file $ENV_FILE -f $COMPOSE_FILE"

mkdir -p "$BACKUP_DIR/postgres" "$BACKUP_DIR/artifacts"

# ─── Postgres ─────────────────────────────────────────────────
DB_USER="$(grep ^POSTGRES_USER "$ENV_FILE" | cut -d= -f2-)"
DB_NAME="$(grep ^POSTGRES_DB   "$ENV_FILE" | cut -d= -f2-)"
DUMP="$BACKUP_DIR/postgres/citrain-$TS.dump"

$COMPOSE exec -T db pg_dump \
  --username="$DB_USER" \
  --format=custom \
  --no-owner \
  --no-privileges \
  "$DB_NAME" > "$DUMP"

# Verify the dump opens.
pg_restore --list "$DUMP" > /dev/null

# ─── Artifacts ────────────────────────────────────────────────
TAR="$BACKUP_DIR/artifacts/artifacts-$TS.tar.zst"
tar --zstd -cf "$TAR" -C /var/lib/citrain artifacts

# ─── Retention (14 dailies each) ──────────────────────────────
ls -1t "$BACKUP_DIR"/postgres/citrain-*.dump 2>/dev/null \
  | tail -n +15 | xargs -r rm -f
ls -1t "$BACKUP_DIR"/artifacts/artifacts-*.tar.zst 2>/dev/null \
  | tail -n +15 | xargs -r rm -f

echo "backup ok: $DUMP $TAR"
```

Install + schedule:

```bash
mkdir -p /opt/citrain/scripts
$EDITOR /opt/citrain/scripts/backup-vps.sh   # paste the script above
chmod +x /opt/citrain/scripts/backup-vps.sh

# Run at 03:30 UTC nightly.
( crontab -l 2>/dev/null; \
  echo "30 3 * * * /opt/citrain/scripts/backup-vps.sh \
        >> /var/log/citrain-backup.log 2>&1" \
) | crontab -
```

## Nightly backup script (mode 2 — LAN beta)

If your LAN-beta install uses Docker named volumes (the default in
`deploy/docker-compose.local.yml`), the artifact bytes live inside
the Docker volume rather than on a host path. Use this variant:

`scripts/backup-lan.sh`:

```bash
#!/usr/bin/env bash
set -euo pipefail

REPO=/opt/citrain
ENV_FILE=$REPO/deploy/env/local.env
COMPOSE_FILE=$REPO/deploy/docker-compose.local.yml
BACKUP_DIR=/var/lib/citrain/backups
TS=$(date -u +%Y%m%dT%H%M%SZ)

COMPOSE="docker compose --env-file $ENV_FILE -f $COMPOSE_FILE"

mkdir -p "$BACKUP_DIR/postgres" "$BACKUP_DIR/artifacts"

DB_USER="$(grep ^POSTGRES_USER "$ENV_FILE" | cut -d= -f2-)"
DB_NAME="$(grep ^POSTGRES_DB   "$ENV_FILE" | cut -d= -f2-)"
DUMP="$BACKUP_DIR/postgres/citrain-$TS.dump"

$COMPOSE exec -T db pg_dump \
  --username="$DB_USER" --format=custom --no-owner --no-privileges \
  "$DB_NAME" > "$DUMP"
pg_restore --list "$DUMP" > /dev/null

# Tar the artifacts volume by attaching a throwaway container that
# can see it. Volume name = <compose-project>_artifacts (compose-
# project defaults to the directory name of the compose file's
# parent, here `deploy`). Confirm at any time with:
#   docker volume ls | grep artifacts
#
# Alpine's busybox tar doesn't speak --zstd; we install GNU tar +
# zstd in the throwaway container so the output is interchangeable
# with the VPS-mode tarballs.
TAR="$BACKUP_DIR/artifacts/artifacts-$TS.tar.zst"
docker run --rm \
  -v deploy_artifacts:/in:ro \
  -v "$BACKUP_DIR/artifacts":/out \
  alpine sh -c "apk add --no-cache tar zstd && tar --zstd -cf /out/artifacts-$TS.tar.zst -C / in"

ls -1t "$BACKUP_DIR"/postgres/citrain-*.dump 2>/dev/null \
  | tail -n +15 | xargs -r rm -f
ls -1t "$BACKUP_DIR"/artifacts/artifacts-*.tar.zst 2>/dev/null \
  | tail -n +15 | xargs -r rm -f

echo "backup ok: $DUMP $TAR"
```

(If you changed the compose project name with `--project-name`, the
volume name changes too — `docker volume ls` to confirm.)

## Off-host replication (mandatory for mode 3, recommended for mode 2)

Local snapshots protect against application-level corruption (a bad
migration, an `rm -rf`). They do not protect against host loss. Push
the contents of `/var/lib/citrain/backups/` off-host every night.

### Option A — `rclone` to an S3-compatible bucket (recommended)

Cheapest at low volume. Backblaze B2, Cloudflare R2, Wasabi all work
for ~$0.005/GB-month. Encrypt **before** upload so the bucket
provider can't read the contents.

```bash
apt install -y rclone
rclone config       # set up an `s3` (or backend-specific) remote,
                    # then a `crypt` remote on top of it.
```

`scripts/replicate.sh`:

```bash
#!/usr/bin/env bash
set -euo pipefail
rclone sync --delete-after /var/lib/citrain/backups crypt:citrain-backups
echo "off-host sync ok at $(date -u +%FT%TZ)"
```

```bash
chmod +x /opt/citrain/scripts/replicate.sh
( crontab -l 2>/dev/null; \
  echo "0 4 * * * /opt/citrain/scripts/replicate.sh \
        >> /var/log/citrain-replicate.log 2>&1" \
) | crontab -
```

Runs at 04:00 UTC, half an hour after the local backup.

### Option B — `rsync` to a second host

```bash
rsync -avz --delete /var/lib/citrain/backups/ backup-host:/srv/citrain-backups/
```

Cheap if you already pay for a second box. Same-provider second box
doesn't protect against provider-wide loss.

### Option C — `borg`

Best space efficiency via deduplication. Heavier to set up.
Recommended once total backup volume crosses ~50 GB.

## Restore procedure

```
                    ┌─────────────────────────────────┐
                    │  How bad is it?                 │
                    └────┬────────────────────┬───────┘
                         │                    │
              "I dropped a row"     "Host is gone"
                         │                    │
                         ▼                    ▼
              Partial restore        Full DR — see below
              (bespoke, work it
              through by hand
              from the dump)
```

### Manual backup-before-something-risky

Before a risky migration or maintenance window:

```bash
/opt/citrain/scripts/backup-vps.sh    # or backup-lan.sh
```

Idempotent — safe to re-run.

### Full DR — restoring onto a fresh host (mode 3)

**Test this end-to-end at least once before you need it.**

> **Critical order-of-operations:**
> 1. Bring up Postgres **only** (step 3).
> 2. `pg_restore` the dump into it (step 4).
> 3. Extract the artifact tarball (step 5).
> 4. **Only then** start the API (step 7).
>
> The api container's entrypoint runs `prisma migrate deploy`. If you
> start it against an empty database, it initializes a fresh schema —
> and then `pg_restore --clean --if-exists` on top of it works, but
> any migrations introduced after the dump was taken will have to be
> re-applied manually because the migrations table now disagrees with
> the schema. **Always restore first; start the API afterward.**

#### 1. Provision a fresh VPS

Walk through `docs/deployment.md` mode 3 steps 1–4 (base packages,
Caddy, firewall, `/var/lib/citrain/` directory tree). **Do not** run
the seed — you'll restore seeded accounts from the backup.

#### 2. Pull the off-host backups onto the new VPS

```bash
rclone sync crypt:citrain-backups /var/lib/citrain/backups
ls -lt /var/lib/citrain/backups/postgres/   # newest first
ls -lt /var/lib/citrain/backups/artifacts/
```

Pick the newest matched pair (timestamps within ~1 minute).

#### 3. Clone the repo and bring up just Postgres

```bash
git clone https://github.com/silvance/Platform.git /opt/citrain
cd /opt/citrain
cp deploy/env/vps.env.example deploy/env/vps.env
$EDITOR deploy/env/vps.env       # use a known password; see note below
chmod 600 deploy/env/vps.env

# Export the DB credentials into the current shell so the rest of
# this section's `pg_restore` / `psql` commands can reference
# $POSTGRES_USER / $POSTGRES_DB without further parsing. `set -a`
# auto-exports every variable defined by the sourced file.
set -a; . deploy/env/vps.env; set +a

docker compose --env-file deploy/env/vps.env \
               -f deploy/docker-compose.vps.yml up -d db
```

> **About `POSTGRES_PASSWORD`**: the dump was taken with
> `--no-owner --no-privileges`, so any password is fine — but **you**
> still need to remember it. A common pattern is to keep the
> `deploy/env/vps.env` off-host alongside the backups (it's a small
> text file). Ensure the bucket itself is encrypted.

#### 4. Restore Postgres

```bash
DUMP=/var/lib/citrain/backups/postgres/citrain-<ts>.dump

docker compose --env-file deploy/env/vps.env \
               -f deploy/docker-compose.vps.yml exec -T db \
  pg_restore --username="$POSTGRES_USER" \
             --dbname="$POSTGRES_DB" \
             --clean --if-exists \
             --no-owner --no-privileges \
  < "$DUMP"
```

Expect non-fatal `NOTICE` lines from `pg_restore`. Fatal errors stop
the process with a non-zero exit.

#### 5. Restore artifacts

```bash
TAR=/var/lib/citrain/backups/artifacts/artifacts-<ts>.tar.zst
tar --zstd -xf "$TAR" -C /var/lib/citrain
```

#### 6. Spot-check: SHA-256 of a random artifact

```bash
docker compose --env-file deploy/env/vps.env \
               -f deploy/docker-compose.vps.yml exec -T db \
  psql -U "$POSTGRES_USER" -d "$POSTGRES_DB" \
  -c "SELECT id, relative_path, sha256 FROM artifacts LIMIT 1;"

# Then:
sha256sum /var/lib/citrain/artifacts/<relative_path>
```

The two hashes must match. If they don't, the dump and the tarball
are out of sync — pick an older pair whose timestamps line up.

#### 7. Bring up the API

```bash
docker compose --env-file deploy/env/vps.env \
               -f deploy/docker-compose.vps.yml up -d --build api
```

Prisma migrations apply on entrypoint. Against a freshly-restored
schema this is a no-op (or applies any new migrations shipped since
the dump was taken).

#### 8. Re-issue Let's Encrypt cert

Point DNS at the new VPS. Caddy fetches a new certificate on the
first request. Watch `journalctl -u caddy -f`.

#### 9. Smoke test

```bash
curl -sS https://api.cicyberlab.com/v1/healthz
curl -sS https://api.cicyberlab.com/v1/readyz   # expect ready:true
```

Sign in via Vercel-hosted frontend with the seed credentials (they
were preserved by the restore). Open a scenario, attempt a question,
view the debrief. If all four pass, the restore is good.

#### 10. Reset seed passwords (precaution)

If the restore took >24 h or the old credentials are in any way
suspect:

```bash
docker compose --env-file deploy/env/vps.env \
               -f deploy/docker-compose.vps.yml run --rm api \
               node dist/scripts/seed.js
```

### Full DR — mode 2 (LAN beta)

Mostly the same as mode 3, with two substitutions:

- Use `deploy/env/local.env` + `deploy/docker-compose.local.yml`.
- Step 5: instead of `tar -x` onto a host path, restore the artifact
  tarball **into the named volume**:
  ```bash
  docker run --rm \
    -v deploy_artifacts:/out \
    -v /var/lib/citrain/backups/artifacts:/in:ro \
    alpine sh -c "apk add --no-cache tar zstd && rm -rf /out/* && tar --zstd -xf /in/artifacts-<ts>.tar.zst -C /tmp && cp -a /tmp/artifacts/. /out/"
  ```

## Log rotation

Both cron jobs append to `/var/log/`. Without rotation they grow
without bound. Drop a small logrotate config:

`/etc/logrotate.d/citrain`:

```
/var/log/citrain-backup.log /var/log/citrain-replicate.log {
    weekly
    rotate 12
    compress
    delaycompress
    missingok
    notifempty
    copytruncate
}
```

`logrotate` runs from `cron.daily` on stock Ubuntu/Debian, so no
extra schedule is needed.

## Monthly restore drill

Calendar reminder: **first Monday of every month, restore the latest
nightly into a throwaway environment**. Two options:

### Option A — Second compose project on the same host

```bash
# Different project name + isolated data root so it doesn't touch
# the live stack.
docker compose --project-name citrain-drill \
               --env-file deploy/env/vps.env.drill \
               -f deploy/docker-compose.vps.yml \
               up -d db
# … pg_restore, extract tar, etc.
docker compose --project-name citrain-drill down -v
```

### Option B — Throwaway VPS at a different provider

Spin up for the drill, shut down afterwards. Also functions as a
paid-for warm spare.

A backup you haven't restored in 3+ months is untested.

## When backups go wrong

| Symptom                                       | First thing to check                                                       |
| --------------------------------------------- | -------------------------------------------------------------------------- |
| `pg_dump` exits non-zero                      | Disk full (`df -h /var/lib/citrain`).                                      |
| `pg_restore --list` complains about the file  | Dump was truncated. Check `pg_dump` exit + disk space.                     |
| Tarball extracts cleanly but file count is low | A bind-mount remount mid-script. Re-run the backup.                       |
| `sha256sum` mismatch between DB and disk       | Dump and tarball taken at different times. Pick a paired set or replay missing artifacts. |
| `rclone sync` complains about auth             | Bucket credentials rotated. Re-run `rclone config`.                        |
| Restored API returns 401 for known passwords   | Restored without `--clean --if-exists`. Re-run with the flags.            |

If any of these happens **during** an incident: keep the corrupt
backup, take a fresh manual one, restore from the next-oldest known-
good pair.
